Prefect artifacts:

- are visually rich annotations on flow and task runs
- are human-readable visual metadata defined in code
- come in standardized formats such as tables, progress indicators, images, Markdown, and links
- are stored in Prefect Cloud or Prefect server and rendered in the Prefect UI
- make it easy to visualize outputs or side effects that your runs produce, and capture updates over time

![Markdown artifact sales report screenshot](/v3/img/ui/md-artifact-info.png)

Common use cases for artifacts include:

- **Progress** indicators: Publish progress indicators for long-running tasks. This helps monitor the progress of your tasks and flows and ensure they are running as expected.
- **Debugging**: Publish data that you care about in the UI to easily see when and where your results were written. If an artifact doesn't look the way you expect, you can find out which flow run last updated it, and you can click through a link in the artifact to a storage location (such as an S3 bucket).
- **Data quality checks**: Publish data quality checks from in-progress tasks to ensure that data quality is maintained throughout a pipeline. Artifacts make for great performance graphs. For example, you can visualize a long-running machine learning model training run. You can also track artifact versions, making it easier to identify changes in your data.
- **Documentation**: Publish documentation and sample data to help you keep track of your work and share information. 
For example, add a description to signify why a piece of data is important.

## Artifact types

There are five user-facing artifact kinds, corresponding to four internal server types (`markdown`, `table`, `progress`, and `image`):

- links (stored internally as `markdown` type)
- Markdown
- progress
- images
- tables

<Note> 
Each artifact created within a task is displayed individually in the Prefect UI.
This means that each call to `create_link_artifact()` or `create_markdown_artifact()` generates a distinct artifact.

Unlike the Python `print()` function (where you can concatenate multiple calls to include additional items in a report), 
these artifact creation functions must be called multiple times, if necessary. 

To create artifacts such as reports or summaries using `create_markdown_artifact()`, define your message string 
and then pass it to `create_markdown_artifact()` to create the artifact.
</Note>


## Server-side behavior

### Artifact keys and ArtifactCollection

When an artifact is created with a non-null `key`, the server upserts an `ArtifactCollection` entry (setting `latest_id` to the new artifact) before inserting the `Artifact` row itself. If the upsert does not result in `ArtifactCollection.latest_id` matching the new artifact, the server raises an error. This means each key has a "latest" pointer that is updated whenever a new artifact is created with that key.

Artifacts created without a key are stored only in the `Artifact` table and will not appear in `/artifacts/latest` results, because they are not tracked in `ArtifactCollection`.

The `/artifacts/{key}/latest` endpoint only consults the `ArtifactCollection` table. If no `ArtifactCollection` entry exists for a key, the endpoint returns 404 and does not fall back to searching `Artifact` rows directly.

### Deleting the latest artifact

Deleting the latest artifact for a key causes the server to update `ArtifactCollection.latest_id` to the next-most-recent artifact (ordered by creation time) and copy that artifact's data, type, description, timestamps, and refs into the collection row. If no other artifacts exist for that key, the `ArtifactCollection` entry is removed entirely.

### Metadata truncation

Artifact metadata values longer than 500 characters are silently truncated per metadata key to 500 characters (with `...` appended) rather than causing a validation error.

### Creating artifacts from result dicts

When creating an artifact from a result dict via `ArtifactCreate.from_result`, keys named `artifact_key`, `artifact_type`, and `artifact_description` are consumed from the dict and mapped to the artifact's `key`, `type`, and `description` fields respectively.

### Updating artifacts

Patching an artifact will also update the corresponding `ArtifactCollection` entry if that artifact is currently the latest (`ArtifactCollection.latest_id` matches the artifact ID). The operation is considered successful if either the `Artifact` or `ArtifactCollection` row was updated.

### Artifact filtering

Artifact filtering applies flow and deployment criteria using `EXISTS` subqueries against `FlowRun` (joining to `Flow` or `Deployment` as necessary) rather than direct joins. This means related-resource filters are evaluated with `EXISTS` semantics.

Filtering the latest-by-key artifacts applies analogous `EXISTS`-based filtering against `ArtifactCollection` rows, using subqueries against `FlowRun` and `TaskRun` when corresponding filters are supplied.

For more information on how to create and use artifacts, see the [how to produce workflow artifacts](/v3/how-to-guides/workflows/artifacts/) guide.
