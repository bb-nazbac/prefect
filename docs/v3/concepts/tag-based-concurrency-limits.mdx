Tag-based concurrency limits prevent too many tasks from running simultaneously by using [task tags](/v3/how-to-guides/workflows/tag-based-concurrency-limits). You can specify a maximum number of concurrent task runs in a `Running` state for tasks with a given tag.

<Note>
As of Prefect 3.4.19, tag-based concurrency limits are backed by [global concurrency limits](/v3/concepts/global-concurrency-limits). When you create a tag-based limit, Prefect automatically creates a corresponding global concurrency limit with the name `tag:{tag_name}`. This implementation detail is generally transparent to users, but you may see these global limits in your UI or API responses.
</Note>

## How tag-based limits work

If a task has multiple tags, it will run only if **_all_** tags have available concurrency.

Tags without specified concurrency limits are treated as unlimited. Setting a tag's concurrency limit to 0 causes immediate abortion of any task runs with that tag, rather than delaying them.

<Tip>
Tag-based task concurrency is different from manually created global concurrency limits, though they can be used to achieve similar outcomes. Global concurrency limits are a more general way to control concurrency for any Python-based operation, whereas tag-based concurrency limits are specific to Prefect tasks.
</Tip>

## Execution behavior

Task tag limits are checked whenever a task run attempts to enter a `Running` state.

If there are no concurrency slots available for any one of your task's tags, it delays the transition to a `Running` state and instructs the client to try entering a `Running` state again in 30 seconds (or the value specified by the `PREFECT_TASK_RUN_TAG_CONCURRENCY_SLOT_WAIT_SECONDS` setting).

## Server-side behavior

### Upsert semantics on creation

Creating a tag-based concurrency limit for a tag that already exists updates the existing internal V2 limit (upsert semantics) and returns a V1-format response. No conflict error is returned for duplicate tags.

### Reset behavior with slot override

When resetting a tag via the reset endpoint, providing `slot_override` replaces active slots with long-TTL leases for the specified task run IDs. Omitting `slot_override` clears all slots for that tag.

### Deletion revokes leases

Deleting a tag-based limit revokes all active leases associated with the limit (in batches) before removing the underlying V2 limit. This immediately frees those slots, which may affect currently running task runs that hold them.

### V1 and V2 semantics differ on increment

The increment handler uses lease-backed atomic acquisition for V2 limits but updates in-memory `active_slots` for legacy V1 limits. These two paths have different side-effect and durability semantics—callers should not assume identical behavior between versions.

For V2 limits, the endpoint attempts atomic bulk acquisition via a DB update. If any requested V2 limit is at capacity, the call fails with 423 for that specific tag.

For V1 limits, availability is determined by comparing the in-memory `active_slots` length to the configured concurrency limit. When at capacity, the endpoint returns 423 with a `Retry-After` header set from `PREFECT_TASK_RUN_TAG_CONCURRENCY_SLOT_WAIT_SECONDS`.

### Zero-limit pre-check

`POST /api/concurrency_limits/increment` rejects the request with 423 Locked if any requested tag has a concurrency limit of 0 (to avoid deadlock), without acquiring other tags' slots.

### V2 active slots from lease storage

`GET /api/concurrency_limits/tag/{tag}` computes the `active_slots` list by querying the lease storage for holders of the underlying V2 limit and filtering for `task_run` holders.

### Merged filter results

`POST /api/concurrency_limits/filter` returns a merged list of V2-converted and V1 limits, deduplicated by tag with V2 preferred. Pagination (`limit`/`offset`) is applied after merging and deduplication.

### V1-format responses from V2 storage

`GET /api/concurrency_limits/{id}` first resolves to the internal V2 limit (if present and named with prefix `tag:`) and maps it to a V1-format response. Otherwise it falls back to legacy V1 records.

### Occupancy tracking on decrement

`POST /api/concurrency_limits/decrement` updates `avg_slot_occupancy_seconds` for V2 limits when `occupancy_seconds` is supplied, using a weighted average over recent samples which influences decay and retry-after behavior.

## Use cases

Tag-based concurrency limits are useful when you have tasks that interact with shared resources. For example:

- **Database connections**: If many tasks across multiple flows interact with a database that only allows 10 connections, you can tag all database tasks with `database` and set a limit of 10.
- **API rate limiting**: Tasks that call external APIs can be tagged and limited to respect rate limits.
- **Resource contention**: Tasks that use memory-intensive operations can be limited to prevent system overload.
